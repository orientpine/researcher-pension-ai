## 1. 감사 요약

### 1.1 감사 범위

206개 투자 가능 펀드(`fund_data.json`) 전체를 대상으로, fund-portfolio 에이전트가 선택한 7개 위험자산 펀드 + 1개 안전자산(예금)에 대해:

1. 각 카테고리의 **모든 경쟁 펀드**를 식별
2. 수익률/수수료 기준 **실제 최적 펀드 여부** 교차 검증
3. 에이전트가 제시한 **대안 비교 범위**의 완전성 검증
4. **데이터 품질 의존 오류** 존재 여부 확인

### 1.2 핵심 결론

| 항목 | 판정 | 설명 |
|------|:----:|------|
| **수치 환각** | ✅ 없음 | 21개 수익률 필드, 8개 수수료 전량 fund_data.json과 일치 |
| **최적 펀드 선택** | ✅ 정확 | 각 카테고리에서 1Y 수익률 기준 최고 펀드를 선택 |
| **전수 비교 제시** | ⚠️ 미흡 | 대안 테이블에 5개만 기재, 10+개 경쟁 펀드 미언급 |
| **데이터 품질 검증** | 🔴 부재 | fund_classification.json 모순(riskLevel=1 + riskAsset=false) 미감지 |

## 2. 발견 사항

### 2.1 [MEDIUM] 대안 펀드 비교 범위 불투명

#### 현상

에이전트가 `99-fund-analysis.md` 섹션 3-2 "대안 펀드 비교"에 기재한 펀드는 **5개**에 불과:

```
미래에셋미국나스닥100인덱스UH
삼성글로벌반도체H
삼성글로벌휴머노이드로봇UH
iM에셋월드골드UH
미래에셋연금한국헬스케어
```

#### 실제 경쟁 펀드 수

| 카테고리 | 존재 펀드 | 대안 테이블 기재 | **미기재** |
|----------|:---------:|:---------------:|:----------:|
| S&P 500 | 3개 | 0개 | 2개 |
| 반도체/AI | 10개 | 1개 (삼성반도체H) | **8개** |
| 로봇/자동화 | 4개 | 1개 (삼성로봇UH) | **2개** |
| 헬스케어 | 3개 | 1개 (미래에셋한국) | **1개** (글로벌헬스케어, -6.46%) |
| 인도 | 5개 | 0개 | **4개** |
| 원자재/금 | 5개 | 1개 (골드UH) | **3개** |

**총 30개 경쟁 펀드 중 5개만 대안으로 제시** → 나머지 20+개는 검토했더라도 보고서에 증거 없음.

#### 영향

- 투자자가 "왜 이 펀드인가?"에 대한 근거를 검증할 수 없음
- 에이전트가 실제로 전수 스캔했는지, 일부만 봤는지 구분 불가

---

### 2.2 [MEDIUM] UH/H 버전 수익률 차이 미명시

#### 현상

유리필라델피아반도체의 UH/H 두 버전:

| 버전 | 1Y 수익률 | 3Y 수익률 | 6M 수익률 |
|------|:---------:|:---------:|:---------:|
| **UH** (환노출) | **66.41%** | **41.04%** | **46.27%** |
| H (환헤지) — 선택됨 | 61.38% | 33.77% | 41.15% |
| **차이** | **-5.03%p** | **-7.27%p** | **-5.12%p** |

에이전트는 H 버전을 선택하면서 "환헤지 50% 밸런스 유지"라고 했으나, **UH 버전과의 수익률 차이(5~7%p)를 명시적으로 비교하지 않음**.

#### 영향

- 환헤지 비용(5~7%p 수익률 희생)의 트레이드오프가 보고서에 부재
- 투자자가 "환헤지를 위해 얼마나 포기하는가?"를 판단할 수 없음

### 2.3 [HIGH] fund_classification.json 데이터 모순 미감지

#### 현상

`iM에셋월드골드증권자투자신탁(주식-재간접형)(UH)` 분류 데이터:

```json
{
  "category": "기타",
  "riskAsset": false,     // ← 안전자산으로 분류
  "riskLevel": 1,         // ← 매우높은위험 (가장 높은 등급)
  "assetClass": "other",
  "themes": ["gold"]
}
```

**`riskAsset: false`(안전자산)과 `riskLevel: 1`(매우높은위험)은 논리적으로 모순.**

- 에이전트는 `riskAsset=false`만 읽고 "안전자산이므로 SAFE_ASSET_DECISION=예금에 의해 불필요"로 기각
- `riskLevel=1`(최고 위험등급)과의 모순을 감지하지 못함

#### 영향 — 시나리오별 분석

**시나리오 A: 골드가 실제로 안전자산(riskAsset=false)인 경우**
- 예금 30% 전체를 예금에 넣을 필요 없이, 예금 20% + 골드 10%로 분배 가능
- 골드 10% 편입 시 안전자산 내에서 수익률 대폭 개선 (4.9% → 혼합 ~19%)
- 에이전트가 이 옵션을 검토하지 않음

**시나리오 B: 골드가 실제로 위험자산(riskLevel=1 기반)인 경우**
- 광업주(96.24%) 대신 골드(145.22%)를 위험자산으로 편입 시 **+49%p 수익 개선**
- 에이전트가 잘못된 분류 데이터에 의존하여 이 기회를 놓침

**어느 시나리오든 현재 포트폴리오보다 나은 대안이 존재했을 가능성 높음.**

#### 근본 원인

`fund_classification.json`의 자동 분류(`source: "fund_data.json + keyword classification"`)가 골드 펀드를 정확히 분류하지 못함. "기타" 카테고리에 넣으면서 riskAsset을 false로 설정한 것은 키워드 매칭의 한계.

## 3. 개선안

### 3.1 [MUST] 전수 비교 증적 (Audit Trail) 의무화

#### 문제
에이전트가 어떤 펀드를 검토했는지 증명할 수 없음.

#### 개선 내용

에이전트 프롬프트에 다음 **Hard Rule** 추가:

```markdown
## HARD RULE: 전수 비교 증적 (MANDATORY)

포트폴리오에 포함하는 **모든 카테고리**에 대해 해당 카테고리의 **전체 경쟁 펀드 목록**을 제시해야 합니다.

### 필수 출력 형식

각 카테고리별로 아래 테이블을 99-fund-analysis.md에 포함할 것:

#### [카테고리명] 전수 비교 (N개 펀드)

| # | 펀드명 | 1Y | 3Y | 6M | 총보수 | 환헤지 | 선택 | 미선택 사유 |
|:-:|--------|---:|---:|---:|------:|:------:|:----:|------------|
| 1 | [선택된 펀드] | X% | X% | X% | X% | H/UH | ✅ | — |
| 2 | [경쟁 펀드A] | X% | X% | X% | X% | H/UH | — | [구체적 사유] |
| 3 | [경쟁 펀드B] | X% | X% | X% | X% | H/UH | — | [구체적 사유] |

### 비교 대상 식별 방법

1. `fund_classification.json`에서 해당 테마/카테고리 필터링
2. `fund_data.json`에서 해당 펀드명 키워드 검색 (보조)
3. 두 방법의 합집합을 비교 대상으로 설정
4. **0개 펀드가 누락되어야 함** — 모든 경쟁 펀드가 테이블에 존재

### 위반 시

전수 비교 테이블이 없는 카테고리의 펀드는 포트폴리오에 편입할 수 없습니다.
```

#### 기대 효과
- 투자자가 모든 대안을 확인 가능
- 에이전트의 선택 근거가 검증 가능해짐
- "일부만 보고 골랐는가?" 의심 원천 차단

### 3.2 [MUST] UH/H 환헤지 비용 명시 의무화

#### 문제
동일 펀드의 UH/H 버전 간 수익률 차이(환헤지 비용)가 보고서에 없음.

#### 개선 내용

에이전트 프롬프트에 추가:

```markdown
## HARD RULE: 환헤지 비용 비교 (UH/H 쌍 존재 시 MANDATORY)

동일 펀드의 UH(환노출)/H(환헤지) 두 버전이 모두 존재할 경우,
반드시 아래 비교 테이블을 포함해야 합니다.

### 필수 출력 형식

#### [펀드명] 환헤지 비용 분석

| 항목 | UH (환노출) | H (환헤지) | 차이 |
|------|:----------:|:---------:|:----:|
| 1Y 수익률 | X% | X% | X%p |
| 3Y 수익률 | X% | X% | X%p |
| 6M 수익률 | X% | X% | X%p |
| 총보수 | X% | X% | X%p |
| **선택** | [O/X] | [O/X] | — |
| **선택 근거** | [사유] | [사유] | — |

### 의사결정 로직

1. UH/H 수익률 차이 = 환헤지의 암묵적 비용
2. 환헤지 비용이 연 3%p 이상이면 경고 표시
3. 환헤지 전략(50%)을 위해 어떤 펀드에서 H를 선택할지 명시적으로 비교
4. "환헤지 비용 vs 환율 변동 리스크" 트레이드오프를 한 문장으로 요약
```

#### 기대 효과
- 투자자가 "환헤지를 위해 얼마를 포기하는가?" 정량적으로 파악
- 환헤지 전략의 비용 대비 효과 투명화

### 3.3 [MUST] 데이터 정합성 교차 검증 (Cross-Validation Gate)

#### 문제
`fund_classification.json`의 `riskAsset`과 `riskLevel`이 모순되어도 감지하지 못함.

#### 개선 내용

에이전트 프롬프트에 추가:

```markdown
## HARD RULE: 데이터 정합성 검증 (MANDATORY — 펀드 선택 전 실행)

### Step 0: Classification 정합성 Gate

포트폴리오 구성 **이전에** 반드시 아래 검증을 수행하세요.

#### 검증 규칙

| # | 규칙 | 조건 | 위반 시 조치 |
|:-:|------|------|-------------|
| 1 | riskLevel ↔ riskAsset 일치 | riskLevel 1-3 → riskAsset=true 필수 | ⚠️ ALERT 출력, 해당 펀드 수동 판정 |
| 2 | riskLevel ↔ riskAsset 일치 | riskLevel 4-6 → riskAsset=false 필수 | ⚠️ ALERT 출력, 해당 펀드 수동 판정 |
| 3 | category ↔ riskAsset 일치 | category="주식형"/"해외주식형" → riskAsset=true | ⚠️ ALERT |
| 4 | category ↔ riskAsset 일치 | category="채권형"/"MMF" → riskAsset=false | ⚠️ ALERT |

#### 검증 실행 코드 (의사코드)

```
for fund in portfolio_candidates:
    classification = fund_classification[fund.name]
    
    # Rule 1-2: riskLevel vs riskAsset
    if classification.riskLevel <= 3 and classification.riskAsset == false:
        ALERT("⚠️ 정합성 오류: {fund.name} — riskLevel={classification.riskLevel} but riskAsset=false")
        MANUAL_REVIEW_REQUIRED = true
    
    if classification.riskLevel >= 4 and classification.riskAsset == true:
        ALERT("⚠️ 정합성 오류: {fund.name} — riskLevel={classification.riskLevel} but riskAsset=true")
        MANUAL_REVIEW_REQUIRED = true
```

#### 위반 발견 시 보고 형식

```markdown
### ⚠️ 데이터 정합성 경고

| 펀드명 | riskLevel | riskAsset | 모순 | 에이전트 판정 |
|--------|:---------:|:---------:|:----:|:------------:|
| [펀드명] | 1 | false | riskLevel 1 = 위험자산인데 false | → riskAsset=**true**로 재판정 |

**근거**: fund_data.json riskLevel이 fund_classification.json riskAsset보다 신뢰도 높음
(riskLevel은 원본 CSV에서 직접 추출, riskAsset은 키워드 자동분류)
```

#### 적용 범위
- 포트폴리오 후보 펀드뿐 아니라, **안전자산 Gate에서 검토하는 채권형 펀드 전체**에도 적용
- 정합성 오류 펀드는 riskLevel 기준으로 재판정 (riskLevel이 원본 데이터이므로 우선)
```

#### 기대 효과
- 분류 자동화 버그로 인한 잘못된 의사결정 방지
- 골드 펀드 같은 "안전자산으로 오분류된 고위험 펀드"의 잘못된 기각 차단
- 데이터 품질 이슈를 에이전트 레벨에서 조기 감지

### 3.4 [SHOULD] 안전자산 다각화 옵션 검토

#### 문제
SAFE_ASSET_DECISION="예금"이면 안전자산 30% 전체를 예금으로 채움. 예금 외 안전자산(riskAsset=false 분류 펀드)과의 혼합 가능성을 검토하지 않음.

#### 개선 내용

에이전트 프롬프트의 안전자산 Gate에 추가:

```markdown
## SAFE_ASSET_DECISION 확장: 혼합 안전자산 검토

### 현행 로직 (유지)
1. 예금 vs bestBond 비교 → threshold 미달 시 "예금" 선택
2. 채권형 펀드 0% 편입

### 추가 검토 (NEW)
3. riskAsset=false인 비채권형 펀드 탐색
4. 해당 펀드가 존재하면 아래 테이블 출력:

#### 안전자산 혼합 옵션 검토

| 구성 | 기대 수익률 | 원금 보장 | 변동성 | 권고 |
|------|:----------:|:---------:|:------:|:----:|
| 예금 100% | 4.9% | O | 없음 | 보수적 |
| 예금 70% + [펀드] 30% | X% | 부분 | 중간 | 균형형 |
| 예금 50% + [펀드] 50% | X% | 부분 | 높음 | 공격형 |

### 의사결정 기준
- 공격형 투자자: 혼합 옵션 적극 검토
- 중립형 이하: 예금 100% 유지 권고
- **단, 정합성 Gate(3.3)을 통과한 펀드만 안전자산으로 편입 가능**
```

#### 기대 효과
- 공격형 투자자에게 안전자산 30%의 수익률 개선 기회 제공
- "안전자산=예금 고정" 사고방식 탈피

### 3.5 [SHOULD] 카테고리 외 고수익 펀드 스크리닝

#### 문제
macro-outlook이 권고한 섹터만 검토하고, 권고 외 고수익 펀드를 무시함.
- 예: 미래에셋연금브라질업종대표 (1Y=56.60%) — 어떤 대안 테이블에도 미등장

#### 개선 내용

에이전트 프롬프트에 추가:

```markdown
## HARD RULE: Top-N 스크리닝 (MANDATORY)

### macro-outlook 섹터 기반 선택 후, 추가 스크리닝 실행

1. fund_data.json 전체에서 **1Y 수익률 Top 20** 펀드를 추출
2. 이미 선택된 펀드와 동일 카테고리가 아닌 펀드를 식별
3. 해당 펀드를 "비선택 고수익 펀드" 테이블로 출력:

#### 비선택 고수익 펀드 (참고용)

| # | 펀드명 | 1Y | 카테고리 | 미편입 사유 |
|:-:|--------|---:|----------|------------|
| 1 | [펀드명] | X% | [카테고리] | macro-outlook 비권고 섹터 |
| 2 | [펀드명] | X% | [카테고리] | 지역 분산 초과 |

### 목적
- 투자자에게 "놓친 기회"가 있는지 투명하게 공개
- macro-outlook 권고의 기회비용을 시각화
- 에이전트가 전체 펀드를 스캔했음을 증명
```
